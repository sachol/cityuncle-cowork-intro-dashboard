## 소개

안녕하세요, 공인중개사님들과 AI 스터디하고 있는 도시아재입니다 🏢. 본업으로 공인중개사 업무를, 그 위에 같은 분야 동료·중개보조원분들을 대상으로 AI 활용 기초 강의를 하고 있습니다.

이 사례는 22기("강사·교육자를 위한 Claude Code + Cowork로 강의 운영 자동화 시스템 만들기") 첫 주에 만든 **OT 전 미니 사례**입니다. v1은 "5분 입문 한 장"을 목표로 만들었는데, 막상 같은 분야 대표님들 입장에서 다시 읽어보니 **"클로드 카우워크가 뭔지", "프로젝트는 어떻게 만드는지", "Instructions는 뭘 적어야 하는지"** 같은 초보 단계가 너무 압축되어 있었습니다. v2는 그 부분을 **5단계 학습 가이드**로 펼치고, 공식 4구성요소와 한계까지 정리해 **"정말 처음 접한 분도 사무실에서 따라할 수 있는"** 톤으로 다시 맞춘 보강본입니다.

그리고 v1에서 가장 큰 인사이트였던 **"기획의도 검증 게이트"** 가 v2에서 **한 번 더 작동했다**는 점이 이 사례글의 핵심입니다. 게이트가 일회성 정정 도구가 아니라 **반복해서 쓸 수 있는 운영 룰**임이 같은 사례 안에서 증명되었습니다.

## 진행 방법

전체 작업은 22기 첫주 수업 후 진행했습니다. 도구는 **Claude Code + Cowork + Tailwind CDN** 만 사용했고, 빌드 도구 없이 **단일 정적 HTML 파일 1개**로 만들었습니다.

> ℹ️ 1~7단계는 v1과 동일한 과정입니다(이력 보존을 위해 그대로 두었습니다). v2의 핵심 보강은 **8단계 — Cowork 학습 가이드 강화 (검증 게이트 2회차)** 에 있습니다.

### 1단계 — 메인·서브 프로젝트 구조 결정

매주 새 결과물을 만들 때 메인 프로젝트가 점점 뒤섞이는 것을 피하려고, 각 주차 사례를 **`subprojects/<name>/` 하위로 물리적으로 분리**하기로 했습니다.

```
gpters22_edu/                       # 메인 — 강의 전체 계획·인프라
└── subprojects/
    └── cowork-intro-dashboard/     # 이번 미니 사례 (서브)
        ├── plan.md
        ├── AGENTS.md
        ├── content.md
        ├── index.html
        ├── case-study.md           # v1
        ├── case-study-v2.md        # v2 (본 문서)
        └── README.md
```

각 서브 프로젝트는 자기 `plan` · `AGENTS` · `README`를 가집니다. 다음 주차 결과물도 같은 패턴(`subprojects/<name>/`)으로 들어갑니다.

> 💡 **이 구조를 첫 사례부터 정한 게 가장 중요한 결정**이었습니다. 결과물끼리 충돌 없이 독립 운영되고, 사례글 작성도 서브 디렉터리 단위로 깔끔합니다. 사례글 버전 관리(v1/v2)도 같은 폴더 안에서 자연스럽게 됐습니다.

### 2단계 — 6 섹션 정보 설계 → 7 섹션으로 확장(v2)

대시보드는 한 장 페이지지만 v2에서는 다음 7 섹션으로 분명히 나눴습니다(원래 6 + PROJECT 신설):

| 섹션 | 의도 | v1→v2 변화 |
|---|---|---|
| HERO | 한 줄 슬로건 + "5단계 학습 가이드" 진입 | 부제에 "5단계 학습 가이드" 명시 |
| WHAT · 뭡니까? | Cowork이 뭔지 어려운 IT 용어 없이 3 카드 | (변경 없음) |
| HOW · 시작하기 | 진짜 5단계 (설치·탭·프로젝트·Instructions·첫 작업) | **3단계 → 5단계** |
| **PROJECT · 구성요소** ⭐ | Instructions / Memory / Context / Scheduled Tasks 4가지 + 한계 4가지 | **v2 신설** |
| APPLY · 적용 | 내 업무 4 진입점 + 시간 절약 칩 | (변경 없음) |
| CASES · 사례 | BEFORE/AFTER 카드 3개 + **Dispatch 시연 (베타)** | (변경 없음, v1에서 정정 완료) |
| NEXT · 다음 단계 | 오늘 안에 해볼 일 3개 체크리스트 | (변경 없음) |

수강생이 모바일에서 한 화면씩 스크롤하며 내려갈 수 있게 구성했습니다.

### 3단계 — Claude Cowork System Instructions (강의에서 그대로 보여줄 패턴)

본문 HOW Step 4에 넣은 Instructions 예시를 그대로 코드블록으로 게시했습니다. v2에서는 v1의 4줄짜리 예시를 **200~500단어 권장**에 맞춰 도시아재 5번 룰로 펼쳤습니다.

```text
나는 서울 ○○구에서 공인중개사로 일하는 "부동산 하는 도시아재"입니다.
주로 다루는 매물: 아파트(재건축 포함), 오피스텔, 상가.
손님은 30~50대 실거주자가 많고, 카톡으로 문의가 들어옵니다.

답변 규칙:
1. 답변은 카톡에 그대로 복사할 수 있는 자연스러운 한국어 존댓말로 해주세요.
2. 시세·면적·금액 등 숫자는 출처가 불확실하면 "확인 필요"라고 표시해 주세요.
3. 매물 광고문은 과장 표현(역대급, 무조건 등) 금지. 사실 기반으로.
4. 분석 결과는 표(시세) + 글머리(강점/약점) + 한 단락(광고 카피) 순서로.
5. 모르는 정보는 추측하지 말고, 어디서 확인하면 되는지 알려주세요.
```

좋은/나쁜 Instructions 비교 박스도 같이 배치해서, "친절하게 답해줘" 같은 모호한 지시가 왜 안 되는지를 보여줍니다.

### 4단계 — Dispatch (베타) 시연 — 📱 폰 → 💻 사무소 → 📲 폰

CASE 04 자리에 Claude Cowork의 **Dispatch (베타, 2026-03-17 출시)** 시연을 넣었습니다. 외근·임장이 많은 공인중개사 업무에 가장 강력한 기능이라고 판단해 꼭 포함하고 싶었던 부분입니다. (v1 7단계의 검증 게이트 1회차로 fan-out 메타포 → 3단 흐름으로 정정한 결과를 그대로 유지)

> *"Message from your phone. Claude works on your computer. You get the result."*
> — Anthropic 공식 (claude.com/product/cowork)

```text
광화문 한일아파트 84㎡, 매매 18억 매물.
오후 임장 예정, 고객은 30대 신혼부부.
시세 분석·광고 카피·임장 직전 카톡 안내·임장 후 보고서 템플릿
4가지 한 번에 만들어서 준비해 둬.
```

> 💡 임장 가는 차 안에서 폰으로 한 줄 → 도착하기 전에 4가지 자료가 준비된 상태. **결과 받으러 사무소에 돌아갈 필요가 없다**는 게 임장·외근이 많은 공인중개사 업무에 가장 큰 가치입니다.

> ⚠️ Dispatch는 베타(research preview)이고 사무소 컴퓨터가 켜져 있고 Claude Desktop이 실행 중이어야 작동합니다. 파일 데이터 추출·메일 요약·보고서 작성은 안정적이고, 일부 서드파티 앱 조작은 아직 불안정합니다.

### 5단계 — 네오브루탈 디자인 시스템 + 라이트/다크 토글

스타일은 굵은 3px 검정 보더 + 6/8px 검정 그림자 네오브루탈로 통일했습니다. 컬러 토큰을 CSS 변수로 정의한 뒤, 다크 모드는 `[data-theme="dark"]` 셀렉터로 변수만 오버라이드해서 다크 토글을 붙였습니다.

```css
:root {
  --ink: #1A1A1A; --paper: #F5F1EA;
  --yellow: #C9A227; --pink: #C97B63;
  --mint: #6F8B6D; --purple: #8A7AA8;
}
[data-theme="dark"] {
  --ink: #E8E4DC; --paper: #1A1815; --card: #252320;
}
```

테마는 localStorage에 저장되고 시스템 `prefers-color-scheme`을 자동 감지합니다. FOUC 방지용 inline script를 `<head>`에 넣어 새로고침 깜빡임도 막았습니다. 처음에는 형광에 가까운 채도였는데 공인중개사 대상이라는 청자에 맞춰 **에디토리얼 머스타드 톤**으로 전면 교체했습니다(저채도·전문가 톤).

### 6단계 — 직접 클릭해보며 다듬기 + 일관 가독성 규칙 확립

미리보기를 켜놓고 직접 클릭해보면서 어색한 점을 발견해 즉시 다듬었습니다. 특히 라이트/다크 양 모드 모두에서 **"흰 배경 안 글씨는 검정, 어두운 배경 안 글씨는 흰색"** 일관 규칙을 CSS 변수와 `:not()` 셀렉터로 자동화했습니다.

```css
[data-theme="dark"] section.bg-white { background-color: var(--paper) !important; }
.bg-white:not(section) {
  background-color: #FFFFFF !important;
  color: #1A1A1A !important;
}
```

라이트 모드에서 잘 보이는 시안이 다크 모드에서는 묻힐 수 있고, 그 반대도 마찬가지였습니다. **두 모드 모두 만든 직후 한 번씩 직접 클릭해보는 것**이 결과물 신뢰의 핵심입니다.

### 7단계 — 게시 직전 "기획의도 검증 게이트" 실행 (1회차, v1에서 도입)

결과물이 거의 완성된 시점, 게시판에 올리기 직전에 작은 한 줄 질문이 던져졌습니다 — *"이 Dispatch가 정확히 뭘 말한 거니?"* 그 한 줄로 게시 전에 4가지를 다시 확인하는 단계가 새로 생겼습니다.

**검증 4개 항목** (이번 일을 계기로 `CLAUDE.md`에 새겨 둠):

1. **기획 의도 일치도** — 결과물 각 섹션이 원래 의도와 일치하는가
2. **사실 정확성** — 외부 도구·기능·플랜·접근 경로를 추측이 아닌 **공식 1차 출처**로 확인했는가
3. **청자 가치** — 청자(공인중개사·동료 강사)가 실제 따라할 수 있는가
4. **본질 흐림 여부** — 작업 중 일반 메타포로 옮겨가서 본 의도가 약해지지 않았는가

**1회차 검증에서 발견된 정정 사항** — 그대로 게시되었다면 첫 단계부터 막혔을 5~6곳:

| 항목 | 정정 전 | 정정 후 |
|---|---|---|
| Cowork Dispatch | fan-out 메타포 | 📱→💻→📲 원격 실행 |
| 가입 경로 | claude.ai 웹 | Claude Desktop 앱 |
| 플랜 | 무료부터 | Pro $20/월 이상 |
| Instructions 길이 | 짧은 한 줄 | 5~8줄 이내 |
| 첫 작업 진입 | 빈 채팅창 | Project 안 새 작업 |

공식 1차 출처를 모두 확인한 뒤 페이지·사례글·README·content·AGENTS를 한 번에 정정해 다시 배포했습니다.

> 📸 [이미지 삽입 위치 3: CASE 04 Dispatch 시연 화면 — 폰→데스크탑→폰 3단 흐름 시각화]

### 8단계 (v2 신설) — Cowork 학습 가이드 강화 (검증 게이트 2회차)

v1을 게시 준비하던 중 추가 검토에서 **"Cowork을 처음 접하는 공인중개사가 프로젝트(폴더)부터 만들어 따라할 수 있도록 학습 가이드 톤이 부족하다"** 는 피드백을 받았습니다. 1회차에서 사실 오류는 잡았지만, **"초보 청자의 학습 동선"** 이라는 의도 일치도 항목은 아직 부족했던 것입니다.

같은 4개 항목으로 다시 점검해봤습니다:

| 항목 | 1회차 결과 | 2회차 점검 |
|---|---|---|
| 기획 의도 일치도 | ✅ 통과 | 🟡 부족 — "초보 공인중개사 학습 가이드"라는 의도에 비해 HOW가 너무 압축 |
| 사실 정확성 | ✅ 통과 (1회차에서 정정) | ✅ 유지 |
| 청자 가치 | ✅ 통과 | 🟡 부족 — Cowork·프로젝트·Instructions 개념을 처음 보는 분에게 불친절 |
| 본질 흐림 여부 | ✅ 통과 | ✅ 유지 |

**공식 출처 1차 재확인**으로 빠진 부분을 매핑했습니다:

- [Get started with Claude Cowork](https://support.claude.com/en/articles/13345190-get-started-with-claude-cowork)

| 항목 | v1 상태 | v2 보강 |

## 결과와 배운 점

### 배운 점

1. 🚨 **검증 게이트는 반복 사용 가능한 운영 룰임이 증명됐습니다.** 1회차(외부 사실 정정)와 2회차(청자 학습 동선 보강)가 같은 4개 항목으로 작동했고, 점검의 초점만 달라졌습니다. **AI로 빠르게 만들수록 한 번의 검증으로는 다 못 잡습니다** — 게시 전 한 번, 다른 시점에 또 한 번 점검하면 매번 새 부족함이 드러납니다. 같이 공부하시는 분들도 본인 결과물 게시 직전·다음 단계 진입 직전에 4개 항목(의도 일치 / 사실 정확성 / 청자 가치 / 본질 흐림) 한 번씩 점검해 보시기를 권합니다.
2. 🎓 **"초보 청자 학습 동선"은 만든 사람 입장에서 가장 잘 안 보이는 항목입니다.** 강사·기획자는 도구를 이미 알기 때문에 "프로젝트 + 버튼 클릭"이 자명해 보이지만, 처음 접하는 분에게는 그 한 줄이 시작 전부의 막막함입니다. **공식 도움말의 단계 라벨(Start from Scratch / Import / Use Existing Folder)을 그대로 살려주는 것**만으로도 학습 가이드 톤이 크게 좋아집니다.
3. 🧩 **첫 사례부터 메인·서브 구조로 분리**한 게 가장 잘한 결정이었습니다. 결과물 디렉터리·문서·사례글이 한 폴더 안에서 깔끔하게 묶이고, **사례글 버전 관리(v1/v2)도 같은 폴더 안에서 자연스럽게** 됐습니다.
4. ✨ **강사가 먼저 직접 적용해 검증**한 결과물 한 장은 슬라이드보다 강한 신뢰가 됩니다. v2에서는 "두 번 검증한 자료"가 됐으니 신뢰가 한 단계 더 쌓였습니다.
5. 🌗 **다크 모드 일관 가독성 규칙은 CSS 변수 + `:not()` 셀렉터로 자동화**할 수 있습니다. `bg-white:not(section)`에 `!important`로 흰 배경/검정 글씨를 강제하면 라이트·다크 양쪽에서 안전합니다.
6. 🔁 **"📱 폰 → 💻 데스크탑 → 📲 폰"** 흐름(Cowork Dispatch 베타)을 한 번 정확히 시연해두면, 외근·임장 중 활용이라는 핵심 가치가 즉시 전달됩니다. W4 크롬 확장 단계에서도 이 컨셉을 그대로 확장할 수 있습니다.

### 시행착오

- **1회차에서 외부 사실 오류 5~6곳** — 자세한 내용은 [v1 사례글](./case-study.md) 또는 위 진행 방법 7단계 참고.
- **2회차에서 청자 학습 동선 부족** (v2 보강의 직접적 동기): "5분 안에 따라하기"라고 적었지만 실제로는 Cowork 모드 진입조차 설명 안 됨. 1회차 검증은 "Dispatch가 뭔지" 같은 사실 항목에 초점이 가 있어서 학습 동선은 놓쳤습니다. **같은 게이트라도 검토자 한 명, 한 번에 모든 것을 잡지 못합니다** — 다른 시점에 다른 초점으로 한 번 더 돌리는 게 게이트의 핵심 가치였습니다.
- **다크 모드에서 흰 배경 박스 안 글씨가 안 보임**: 단순 색 지정으론 부모 카드 자식 선택자가 이긴 케이스가 있었습니다. CSS 우선순위(specificity) 차이를 `!important` + `:not()`으로 잡았습니다.
- **에디토리얼 머스타드 톤으로 팔레트 전면 교체**: 처음 형광에 가까운 채도는 공인중개사라는 청자 톤과 어울리지 않아 저채도 전문가 톤으로 일괄 교체. 청자가 누구냐에 따라 디자인 톤도 검증 항목입니다.

### 앞으로의 계획

이 미니 사례는 22기 4주 마일스톤의 첫 주에 만든 **"강의 본 사례 옆에 두는 1장 견본"** 입니다. 다음 회차들에서는 본 사례·강의자료·대시보드 결과물이 차례로 이어질 예정이고, 그때마다 사례글로 따로 공유하겠습니다. **검증 게이트는 W2~W4 모든 결과물에 자동 실행**됩니다.

가까이는, **이 미니 사례를 다음 회차에 강의자료 1차시로 변환**해보면서 — 한 장 페이지가 어떻게 슬라이드·워크북·퀴즈 1세트로 자연스럽게 풀리는지를 확인할 계획입니다.
